Description

In the previous stage, we implemented user registration and handled some errors that users might make. However, the app should not crash if the user makes a mistake.

In addition, we need to store customer data. In addition to registration, we will also add authorization. We must give users clear instructions and error messages to create a quality application. For example, we should not allow users to enter cabbage in the login field. Therefore, we will need to standardize user input. We will learn how to filter the customer data so that each customer is unique, which is standard practice in banking.

How a database might look in your file:

    Login: +1 231 233 3333, Password: Secret98 password, Registration time: 2023-05-31 15:30
    Login: +1-231-233-4444, Password: Mysecurepassword123, Registration time: 2023-05-31 11:32
    Login: +1-231-233-5555, Password: P@ssw0rd!, Registration time: 2023-05-31 09:15
Objectives
  1. Submit an authorization form Login: Password:
  2. Let's change the Username field to Login, and now the user will enter his personal phone number, which allows you to make each client unique. HashMap will help us with this.
  3. We will store the client database in a file you create using the File functionality or directly in the development environment.
  4. If a user with this login already exists in the database, display the message Login is already taken, try another login. And then ask for input again.
  5. The first time you successfully enter data, your program should request them again, displaying this message: Now you can log in for finishing the registration.
  6. To finish registration user should be able to log in and enter the login and password. After that, the program should display the two messages: Authorization successfuland Congratulations on your successful registration!
  7. If you failed to authorization, no matter whether the password or login was entered incorrectly, you should display the message: Authorization failed. After that, your program should end

Hint: when implementing the method of checking the login for uniqueness, don't forget to consider the same parameters; for example, these numbers are the same: +1 231 233 3333 and 12312333333. The devil is in the phones and the details.

Another hint: If the user received a message about successful registration, it would be strange if the data was not saved in the database.

The phone number format must follow the USA regional number standard:

  • The phone number must only start with the number 1.
  • The phone number can contain the + sign.
  • The phone number can be without the + and - signs, for example, X XXX XXX XXXX or XXXXXXXXXXX.
  • The number must only be 11 digits long.
  • If the user encloses a group of digits in parentheses, only one group can be enclosed in parentheses in the phone number.
    For example, (X)-XXX-XXX-XXXX is allowed while X-XXX-(XXX)-XXXX is invalid.
  • The phone number should not have a space at the beginning or end.
  • The allowed characters in the phone number are:
    • + can only be at the beginning of the number.
    • - serves as a separator between groups of digits.
      • Should not be at the beginning or end of the number.
      • Can separate a group of digits consisting of a single number. Example: X-XXX...
      • Can separate a group of digits consisting of three numbers. Example: X-XXX-XXX...
      • Can separate a group of digits consisting of four numbers. Example: X-XXX-XXX-XXXX
      • Can be replaced with a space while adhering to the aforementioned requirements.
    • The number can contain only four groups of digits X XXX XXX XXXX or one group of digits XXXXXXXXXXX separated. X XXX XXXXXXX is wrong.

Password:

The password should meet the following requirements:

  1. It must contain at least one uppercase letter, one lowercase letter, and at least one digit.
  2. It may contain special characters such as !, @, #, $, %, or not.
  3. It can include spaces, but not at the beginning or end of the password.
  4. It should consist only of Latin alphabet letters.
  5. The length of the password should be at least 6 characters and not exceed 28 characters.
  6. In case of incorrect login or password input, it is necessary to prompt for re-entry and display a warning about incorrect data format. Invalid Login: Wrong login format, try again. Invalid Password: Wrong password format, try again

For testing purposes, create a file in the main project folder called userData.txt; your database must store and read data from the specified file.
